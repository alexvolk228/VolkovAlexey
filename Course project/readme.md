### 9. Подготовка к интерпретации построенных моделей

**9.1 Используемые паттерны проектирования и разработки [✋](https://github.com/stankin/design-part-2/wiki/sem2):**

* **Процессная модель для сравнения:**

Задача: создание системы для наиболее эффективного взаимодействия с клиентами.

**Решение задачи при помощи методологии PDCA:**

1. **Plan (Планирование):**

 + Выявленные проблемы :

   + неэффективная работа с заказами;
   + отсутствие истории взаимодействия с клиентами;
   + аналитика продаж и её визуализация занимает много времени;

 + Цели: 

   + сокращение временных затрат на обработку заказов;
   + ведение истории взаимодействия с клиентами;
   + автоматическое создание отчётов. 

 + Требования: 
   + повысить эффективность без увеличения трудозатрат;
   + конечная система должна быть проста в использовании, как для администратора, так и для остальных сотрудников.

 + Ожидаемый результат: CRM-система с функционалом, позволяющим обеспечивать эффективное взаимодействие с клиентами.

 + Ресурсы, необходимые для достижения ожидаемого результата: 

   + CRM-система;
   + БД;
   + Веб-сервер;
   + Сервер. 

 + Процессы (запланированные действия):

   + Проектирование;
   + Верстка;
   + Настройка и подключение БД;
   + Fullstack - разработка системы;
   + Ручное тестирование. 

2. **Do (Выполнение)** : Выполнение поставленных задач сотрудниками, обладающими правами на использование системы. 

3. **Check (Проверка)** : Оценка эффективности на основе результатов внедрения и использования системы. 

4. **Act (Улучшение)** : При успешной работе системы (при сокращении временных затрат на обработку заказов), он внедряется в информационную систему предприятия, с возможностью доработки или расширения функционала в дальнейшем. 

* **Используемые в разработке паттерны и фреймворки:**
  + Фреймворк: Materialize CSS

**9.2 Используемые паттерны выявления проблем [✋](https://github.com/stankin/design-part-2/wiki/sem3):**
* Муда: создание отчётности вручную. 
* Мура: обработка всех заказов в конце недели.

**9.3 Возможные антипаттерны [✋](https://github.com/stankin/design-part-2/wiki/sem4):**

| Категория  | Антипаттерн (риск) | Действие по избежанию |
|---|---|---|
| Разработка | Таинственный код (Cryptic code) - использование аббревиатур вместо понятных имен | Задания имён переменным и функциям, в соотвествии с их назначением |

**10.2 Определение числовых показателей для цели потенциального проекта автоматизации: [✋](https://github.com/stankin/design-1/wiki/interpret_process)**

Менеджер тратит:
  + Получение заказа 10 мин
  + Передача заказа на производство 5 минут
  
Среднее число заказов в день - 100. Следовательно, одному менеджеру нужно иметь рабочую смену в 25 часов для обработки всех заказов, когда рабочая смена должна быть 8 часов. Для успешной обработки такого количества заказов потребуется, как минимум, 4 сотрудника.

**10.3 Расчет потенциального эффекта от автоматизации:**

Один менеджер за смену (8 часов) может обработать = 32 заказов.

После внедрения системы управления взаимоотношениями с клиентами, на обработку одного заказа будет тратиться 30 секунд на получение заказа интернет-магазина и проверку его корректности + 1 минута на передачу заказа на производство. Итого - 1,5 минуты. Экономия составляет 13,5 минут с одного заказа, или 22,5 часа в день на одного менеджера.

С автоматической системой менеджер сможет обработать за смену = 320 заказов, что в 10 раз больше. А при суточном потоке в 100 заказов, с данной работой справится всего 1 менеджер.

**10.4 Определение числовых показателей и расчет трудозатрат на разработку программных средств:**

Расчет сложности разработки методом FPA IFPUG:
![FPA](https://github.com/alexvolk228/VolkovAlexey.github.io/blob/master/Course%20project/Screenshots/FPA.png?raw=true)

Расчет трудозатрат на разработку «с нуля» методом COCOMO II:
![COCOMO](https://github.com/alexvolk228/VolkovAlexey.github.io/blob/master/Course%20project/Screenshots/COCOMO.png?raw=true)
**ВЫВОДЫ**

Исходя из расчётов трудозатрат, полученных методом COCOMO II делаем вывод, что система является экономически выгодной (с учетом того, что программисты имеют низкие навыки) и может быть реализована менее чем за 9 месяцев командой из 25 человек.

После внедрения CRM-системы, при количестве заказов в день равным 100 экономия времени составит 22,5 часа в день на отдел менеджеров из 5-х человек, это позволит иметь в штате меньшее количество сотрудников, тем самым снизятся затраты на зарплаты.

Плановое время, затраченное на разработку системы, составляет 9,1 мес, при трудоемкости 24,3 ч/мес. Создание системы: 9,1 мес = 17 496 ч/час на разработку.

В день на одного менеджера экономия составляет 22,5 часов, то есть в день отдел из 5-ти менеджеров экономит 112,5 часов. При 5-ти дневной рабочей неделе энономия в месяц составит 2250 ч/час. Значит внедрение системы будет иметь точку безубыточности примерно через 8 месяцев после даты начала использования.

В результате расчета сложности разработки методом FPA IFPUG было получено 3264 строк кода. На данный момент система разработана на 50% и имеет 1632 строки кода. Снижение строк кода возможно только в минимальном количестве. Трудозатраты были снижены за счет упрощения документирования и использования предыдущих наработок. Определен риск срыва сроков. При срыве сроков разработки будут сокращены требования для текущей версии системы, тем самым снизятся трудозатраты на разработку и количество строк кода.
